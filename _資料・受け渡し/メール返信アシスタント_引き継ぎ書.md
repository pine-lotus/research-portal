# メール返信下書き自動化ツール ― 設計・運用ルール（引き継ぎ書）

> このファイルは、クライアントのPC上のClaude Codeに読ませるための引き継ぎ書です。
> 開発の前提・設計思想・守るべきルールをまとめています。**まず全体を読んでから着手してください。**

---

## 1. このプロジェクトの目的

POP3でメールを受信しているユーザーのために、「特定の相手からのメールに対する**返信の下書き**」を自動で作るツールを作る。

- 最終的に使うのはクライアント本人（非エンジニア）。
- ツールの「頭脳」にあたる返信ルール（`SKILL.md`）は、開発担当者が遠隔でチューニングし続ける。
- コードは GitHub の**非公開リポジトリ**で共有し、開発者が `push`、クライアントが `pull` して最新化する。

---

## 2. 絶対に守るルール（最重要）

### セキュリティ
- **メールのパスワードやPOP3接続情報を、絶対にGitに上げない。** 秘密情報は `.env` に分離し、`.gitignore` で除外する。
- Gitで共有するのは「コードと返信ルール（文面ロジック）」だけ。秘密情報は各PCローカルにのみ置く。
- クライアントのメールは第三者の機密情報。**本人の明確な同意のもと**で扱う。

### 自動化の範囲
- **「下書きまで」で止める。自動送信（SMTP送信）はしない。** 送信は必ず人間が確認してから手動で行う。
- 誤送信・失礼な返信は取り返しがつかないため、人の確認を必ず挟む。

### 進め方
- いきなり全部作らず、**小さいステップで進める**（例：まず「指定の相手の新着メールを受信するだけ」から）。
- 各ステップごとに動作確認してから次へ。
- 「ある修正で他が壊れるリスク」がある場合は、作業前に必ず確認する。

### 説明の仕方（相手は非エンジニア）
- 専門用語には必ず平易な補足を入れる。
- コードの各ステップが「何をしているか」と「なぜそうするか」を日本語で説明する。
- エラー時は必ず【原因】と【対処法（具体的な手順）】をセットで示す。

---

## 3. 技術的な前提と制約

- 受信は **POP3**（メールを受信＝ダウンロードするだけの仕組み。送信はできない）。
- **POP3にはサーバー上の「下書きフォルダ」が無い。** よって「下書き」は、サーバーではなく**ローカルにファイル（`.eml` またはテキスト）として書き出す**形で実現する。
- 出力した下書きは、クライアントが普段使っているメールソフト（Outlook / Thunderbird / Macメール 等）で開いて確認し、手動送信する。
- 実行環境：クライアントのPC上の **Claude Code**。Skill（`SKILL.md`）として提供する。
- 実行タイミング：毎朝など**定期実行**を想定（Claude Codeのスケジュール機能。PCが起動している時間に動く）。

---

## 4. 全体の動き（完成イメージ）

```
毎朝 指定時刻に、クライアントPCのClaude Codeが実行
   ↓
① POP3でメールサーバーに接続し、新着メールを受信
   ↓
② 「指定した相手」からのメールだけ選別
   ↓
③ メール本文をClaudeが読んで返信文を作成（トーン・定型文は SKILL.md に従う）
   ↓
④ 返信文を「下書きファイル(.eml/テキスト)」としてローカルに保存
   ↓
⑤ クライアントが確認 → 自分のメールソフトで手動送信
```

---

## 5. リポジトリ構成（設計図）

```
mail-reply-assistant/
├─ README.md                  クライアント向けの導入・使い方手順書（平易な日本語）
├─ .gitignore                 秘密情報をGitに上げないための除外設定
├─ .env.example               接続情報の記入例（中身は空。実物は各PCで .env を作る）
├─ .claude/
│   └─ skills/
│       └─ mail-reply/
│           └─ SKILL.md       ★返信ルールの本体（対象の相手・トーン・定型文）。開発者がチューニング
├─ scripts/
│   ├─ fetch_mail.py          POP3で新着メールを受信する処理
│   └─ save_draft.py          返信文を下書きファイル(.eml等)に保存する処理
└─ drafts/                    出来上がった下書きの保存先（Git管理外）
```

**設計の肝**：「変わる部分（`SKILL.md`）」と「変わらない部分（スクリプト）」を分離する。
- 開発者は基本 `SKILL.md` だけを調整する。
- クライアントは初回に `.env` を記入するだけ。

---

## 6. 役割分担

| 項目 | 担当 |
|---|---|
| 返信トーン・対象・定型文（SKILL.md）のチューニング | 開発者（push） |
| POP3接続情報・パスワードの記入（.env） | クライアント本人（自分のPC） |
| 日々の実行・下書きの確認・送信 | クライアント本人 |
| コードの最新化 | クライアントが pull |

---

## 7. 最初の一歩（推奨ステップ）

いきなり全機能を作らず、次の順で進める：

1. **箱を作る**：リポジトリと最小ファイル（README / .gitignore / .env.example）だけ用意。
2. **受信だけ**：POP3で「指定の相手からの新着メールを受信して表示するだけ」を作り、接続確認。
3. **下書き生成**：受信したメールから返信文を作る。
4. **下書き保存**：返信文を .eml/テキストで書き出す。
5. **定期実行**：毎朝動くようスケジュール設定。

各ステップで必ず動作確認し、問題があれば原因と対処法をセットで説明すること。

---

## 8. 着手前に確認すべき未確定事項

- クライアントが使っているメールソフト（下書きの出力先・形式を決めるため）。
- POP3の接続情報（サーバー名・ポート・SSL有無・メールアドレス）。※パスワードは .env で安全に扱う。
- 下書きの確認形式の好み（.eml ファイル / テキスト・Markdown一覧）。
