# CLAUDE.md — ゾンタック AI導入プロジェクト

## ⚠️ 絶対に守るルール：ダブルブッキング（Wブッキング）完全禁止

**同じ部屋・同じ日程に2つ以上の予約を確定させてはならない。これは業務上の最重要ルール。**

- 予約確定処理（`confirmReservation()`）では、割り当て済み全部屋 × 全宿泊日を、他の確定済み予約と必ず照合すること
- 1件でも重複が検出されたら、確定を**完全にブロック**する（強制確定・例外なし）
- 重複が検出された場合は「○○室が ○月○日〜○日に【団体名】と重複しています」と具体的に表示する
- 部屋割り自動提案（`autoAssignRooms()`）は、確定済み予約で使用中の部屋を除外した上で提案すること（「未割当」が出ないよう空き部屋を必ず具体的に指定する）
- 重複が生じた場合のフロー：該当団体にメールで現状を伝え、別の日程・部屋を提案する（自動送信はしない。下書きまで）

## このプロジェクトについて

- 菅平のプチホテル「ゾンタック」のフロント・バックオフィス業務を、AIで効率化する。
- 最初の対象業務：**空室の見える化 ＋ 予約・問い合わせの返信下書き**（自動送信はしない）。
  - 背景：紙＋スプレッドシートで「先々の空室が一目で分からない」ため、問い合わせ返信と予約入力に時間がかかっている（最大の課題）。
- ユーザー（依頼者）は**非エンジニア**。専門用語には必ず平易な補足を入れ、各ステップが「何をしているか」「なぜそうするか」を日本語で説明すること。
- 守るべきルールの詳細は `開発ルール.md`、宿の基本情報・プラン・FAQは `宿情報.md` を参照する。

---

## 守るべき開発ルール（最重要・必ず守る）

詳細は `開発ルール.md` を参照。要点は以下：

1. **セキュリティ**：開発時は常にセキュリティ上の問題がないか確認する。問題があれば作業を止め、**都度ユーザーの承認**を得る。
2. **個人情報はクラウドに上げない**：宿泊客の氏名・連絡先・予約内容・メール本文などは**現地PCのみに保存**する。Git push・外部API送信・外部保存をしない。
3. **作る前に立ち止まる**：コードを変更する前に、必ず**仕様を検討し、リスクと代替案を提示**し、**ユーザーの承認を得てから**進める。
4. **完了前に自己確認**：開発が終わったら、まず**動作と全体のコードに問題やエラーがないかを自分で確認**し、問題がないことを確かめてから完了報告する。
5. **自動送信しない**：メール等は当面「下書き」まで。人が確認してから送信する。

## データの取り扱い（線引き）

| 種類 | 置き場所 | Git |
| --- | --- | --- |
| コード・指示書 | 共有 | ✅ 上げる |
| 実データ（予約・顧客） | 現地PCだけ | ❌ 上げない |
| 詳細ログ（個人情報あり） | 現地PCだけ | ❌ 上げない |
| 安全ログ（値は伏字／場所・種類・形は残す） | 共有 | ✅ 上げる |

- リポジトリは必ず **Private**。
- **安全ログの作り方**：個人情報の「**値**」は伏字（`***`）にする。ただし**エラーの場所・種類・データの"形"（件名の有無／本文の文字数／日付の書式／添付の有無 など）は残す**（伏せすぎるとリモートで修正できなくなるため）。必要なら「値だけ伏字にしたマスク済みサンプル」を1件だけ残してよい。

## 開発・運用の環境

- PC：**Windows 11**（フロントの常時起動PC・常時ネット接続）。
- ツール：Claude Code ／ Git（GitHub・Private）。
- リモートデスクトップは使わない。このPCのコードは **Git 経由で更新する**（ユーザーが遠隔から push したものを、このPCが pull で取り込む）。
- 実行は「**ボタン（.bat ファイル）**」で行う（Windowsなのでダブルクリックで動く .bat）：
  1. 最新コードを取り込む（git pull）
  2. 処理を実行する
  3. 安全ログを Git に上げる（git push）

## 現状の業務・環境（記入済ヒアリングシートより）

- 予約の管理方法：**紙 ＋ スプレッドシート**（先々の空室が一目で分からないのが課題）
- 宿泊管理ソフト：「リバティ あすなろ」（チェックイン／アウト管理に使用）
- メール環境：**Outlook**
- 予約サイト(OTA)：じゃらん・楽天（ただしOTA経由の予約は少なめ）
- 会計・経理：マネーフォワード（MF）を使用。勤怠もMFへ移行したい意向
- PCの操作者：**社長・副社長**
- 個人情報の方針：効率化のためのツールとして歓迎。セキュリティも重視するが、必要以上のリスクヘッジは不要。
- 着手業務：**空室の見える化 ＋ 問い合わせ返信下書き**（大変さ4の課題を同時に解消）

## 話し方・進め方のルール

- 非エンジニア向けに、専門用語は平易に説明する。
- エラーが出たら、**「原因」と「対処法（具体的な手順）」をセット**で説明する。
- 大きな変更は、いきなり作らず、**全体像を先に提示**してから進める。
- 一度に大量に作らず、小さなステップで進める。
